Buttons

A button is a fundamental element of an interface that allows users to make choices on a journey. They have the power to compel users to convert or carry out an action on a page. The copy used on the button should be clear, concise and descriptive.

Primary button

Primary buttons are used for high emphasis call to actions. Ideally, there should only be one primary call to action button per page.

Open in new window
<button type="submit" class="btn btn--primary">Primary button</button>

Secondary buttons

Secondary buttons are the alternaties we give users to primary actions. Secondary buttons are the ‘go back’ to the primary button’s ‘next’, or the ‘cancel’ button to the ‘submit’ button.

Secondary buttons can also be used for supplementary actions which users take. They can indicate a positive action that isn't necessarily the main call-to-action on a page.

Outlined

Open in new window
<button type="button" class="btn btn--outlined">Outlined button</button>

Filled

Open in new window
<button type="button" class="btn btn--filled">Filled button</button>

Neutral

Open in new window
<button type="button" class="btn btn--neutral">Neutral button</button>

Call to action

The call to action button should be used when a link is the primary call to action on a page.

Open in new window View report
<a href="#" role="button" class="btn btn--primary btn--link">View report</button>

Disabled button

Use with the disabled="true" element attribute for buttons that have no event/action but need to be displayed.

Open in new window
<button type="button" class="btn btn--primary-disabled" disabled="true">Disabled button</button>

Button group

Group two or more buttons together and align them alongside eachother. This will allow for a page to have two actions available to the user such as Continue and Cancel.

Open in new window
<div class="btn-group"> <button type="button" class="btn btn--primary btn-group__btn">Continue</button> <button type="button" class="btn btn--outlined btn-group__btn">Cancel</button> </div>

Icon and text button

Icons work well within buttons when they represent something functional such as 'download' or 'print'. They can clarify an action and call attention to a button.

Open in new window
<button type="button" class="btn btn--filled btn--print">Print this report</button>

Design guidelines

Always emphasise on the primary action

Visually differentiate the primary action by making it more prominent – never use two competing primary buttons next to each other.

Don’t have more than one primary action on a page at a time

Let users focus on the task at hand. You can have numerous actions on a page, but make sure there's only one primary action.

If you have more than one competing primary action, use secondary buttons.

Left align the primary button on full page designs

If the browser window is large, or you have to scroll to fill out a form, always place the primary button right below the last bit of content on the page. The reason is that this is where your users' attention has been focused, while either reading or adding information to a form.

Place the primary action bottom right in dialog boxes

In smaller windows it’s best practice to have the most important action at the end.

Compare the scanning patterns below to see which is easiest – hence, the primary action should be bottom right.

Ensure button labels fit on one line

Don’t wrap text. For maximum legibility, a text label should remain on a single line.

Add clear text labels to buttons

The labelling of buttons matter. Make sure to clearly explain what your button will do, once clicked.

Accessibility

Buttons by default use the button element, which is appropriate for triggering actions within the current page. If the control is being used as a navigation action (i.e. the page address will change as a result of clicking the button) then an a href element should be used.

Disabled buttons

Use the disabled=true or aria-disabled=true attributes to convey information to screen readers (only use aria attributes as a last resort)

Add cursor: not-allowed to the disabled styles, this will serve as another visual indicator that the button is disabled for mouse users.

Avoid disabled submit buttons

There are a number of usability and accessibility issues with disabling form submit buttons until the user has entered valid/expected data. It's often better to let the user attempt submission and provide explicit feedback in response.

Icon buttons

Whenever possible, provide a text description as well to ensure the best level of understanding is conveyed. If an icon is used on its own, make sure screen reader only text is provided using the class visually-hidden so a description is provided to users with assistive technologies.
